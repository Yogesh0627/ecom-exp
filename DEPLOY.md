# Deploying EcoExpress to Render

Two services (API + storefront) on Render, plus an **external hosted Postgres** you create. Object
storage (Cloudflare R2) and email (Resend) are already cloud services and need no deployment.

Everything is env-var driven — no code changes to deploy. The steps below are ordered so each one
has the values the next one needs.

---

## 1. Create the database (you do this)

Render's own free Postgres expires after 90 days, so use a persistent free host — **Neon**
(neon.tech) is a good fit (free, Postgres 16, doesn't expire).

1. Create a Neon project → it gives you a connection string like
   `postgresql://user:pass@ep-xxx.ap-southeast-1.aws.neon.tech/dbname?sslmode=require`
2. Convert it to the three values Render needs. **Two gotchas** turning a Neon `postgresql://`
   string into a JDBC one:
   - Use the **direct** endpoint, not the pooler — drop `-pooler` from the host. A JDBC app has its
     own connection pool (HikariCP); Neon's PgBouncer pooler breaks pgjdbc's prepared statements.
   - Drop `&channel_binding=require` — that's a libpq option pgjdbc doesn't understand (it errors on it).

   So `postgresql://user:pass@ep-xxx-pooler.region.aws.neon.tech/db?sslmode=require&channel_binding=require`
   becomes:
   - `DB_URL` = `jdbc:postgresql://ep-xxx.region.aws.neon.tech/db?sslmode=require`  *(jdbc: prefix, no `-pooler`, keep `sslmode=require`)*
   - `DB_USERNAME` = the user
   - `DB_PASSWORD` = the password

Flyway runs all migrations automatically on the API's first boot — no manual SQL. (Verified
against Neon PostgreSQL 18: all migrations apply; Flyway logs a harmless "newer than tested"
warning.)

> The catalog is **empty** on a fresh DB. The bootstrap admin is created automatically; a
> pre-verified demo customer is seeded too (see demo login below). Add products via the Admin
> console (or your own seed) after first boot.

## 2. Deploy the services

The two apps are deployed to **different platforms**, each the better fit:

| App | Platform | Why |
|---|---|---|
| **Backend API** (`server/`) | **Render** (Docker) | Long-running JVM service |
| **Frontend** (`client/`) | **Vercel** | Native Next.js host — edge CDN, image optimisation, zero Docker |

> ⚠️ **Do not deploy the frontend on Render.** `render.yaml` intentionally defines **only** the API
> service. Building Next.js in Docker on Render's free tier is slow and fails easily, and it would
> duplicate what Vercel already does better.

### 2a. Backend → Render

**Option A — Blueprint (recommended).** Push this repo to GitHub, then in Render →
**New → Blueprint** → point it at the repo. It reads `render.yaml` and creates `ecoexpress-api`.
It will prompt for every `sync: false` env var — you can fill them now or after (some need the
frontend URL from step 2b).

**Option B — Manual.** Create one **Web Service** from the repo with **Runtime = Docker**:
- Root Directory `server`, Dockerfile `server/Dockerfile`, Health Check Path `/actuator/health`

**Free** plan, region **Singapore** (closest to India).

### 2b. Frontend → Vercel

In Vercel → **Add New → Project** → import the repo, then:
- **Root Directory:** `client`
- **Framework Preset:** Next.js (auto-detected). Leave build/output settings default.
- Add the `NEXT_PUBLIC_*` env vars from step 3, then **Deploy**.

You'll get a URL like `https://ecoexpress-nu.vercel.app`.

## 3. Set environment variables

After both exist you know their URLs (e.g. API `https://ecoexpress-api.onrender.com`, frontend
`https://ecoexpress-nu.vercel.app` — use whatever each platform actually assigns).

**On `ecoexpress-api`:**

| Variable | Value |
|---|---|
| `DB_URL` / `DB_USERNAME` / `DB_PASSWORD` | from step 1 |
| `BOOTSTRAP_ADMIN_EMAIL` / `BOOTSTRAP_ADMIN_PASSWORD` | your first admin login |
| `APP_BASE_URL` | the **API** URL, e.g. `https://ecoexpress-api.onrender.com` |
| `CORS_ALLOWED_ORIGINS` | the **Vercel** URL, e.g. `https://ecoexpress-nu.vercel.app` — **no trailing slash** (the origin is matched exactly) |
| `OAUTH_SUCCESS_REDIRECT` | `https://ecoexpress-nu.vercel.app/auth/callback` |
| `OAUTH_FAILURE_REDIRECT` | `https://ecoexpress-nu.vercel.app/login` |
| `EMAIL_VERIFY_URL` | `https://ecoexpress-nu.vercel.app/verify-email` |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | from your Google OAuth client |
| `GEMINI_API_KEY` | Gemini key (AI features) |
| `RESEND_API_KEY` | Resend key (email) |
| `EMAIL_FROM` | `Eco Express <noreply@mail.yogeshchauhan.dev>` |
| `S3_ENDPOINT` / `S3_BUCKET` / `S3_ACCESS_KEY` / `S3_SECRET_KEY` / `S3_PUBLIC_URL` | your R2 values |
| `RAZORPAY_KEY_ID` / `RAZORPAY_KEY_SECRET` | Razorpay keys |
| `RAZORPAY_WEBHOOK_SECRET` | from step 5 (fill after creating the webhook) |
| `INVOICE_SELLER_*` | your registered business details (GSTIN, FSSAI required) |

`JWT_SECRET` is auto-generated by Render — leave it. `STORAGE_PROVIDER=s3`, `S3_REGION=auto`,
`AI_MODEL=gemini-flash-latest` are preset in `render.yaml`.

**On Vercel (frontend)** — Project → Settings → Environment Variables:

| Variable | Value |
|---|---|
| `NEXT_PUBLIC_API_BASE_URL` | the **API** URL **+ `/api/v1`**, e.g. `https://ecoexpress-api.onrender.com/api/v1` |
| `NEXT_PUBLIC_SITE_URL` | the **Vercel** URL, e.g. `https://ecoexpress-nu.vercel.app` (canonical/OG/sitemap) |
| `NEXT_PUBLIC_PAYMENTS_TEST_MODE` | `true` while on Razorpay test keys (shows the test-card hint); `false` once live |
| `NEXT_PUBLIC_ENABLE_DEMO_LOGIN` | `true` to show the one-click "As Admin / As User" buttons |
| `NEXT_PUBLIC_DEMO_ADMIN_EMAIL` / `NEXT_PUBLIC_DEMO_ADMIN_PASSWORD` | must match the API's `BOOTSTRAP_ADMIN_*` so the "As Admin" button works |

> **All `NEXT_PUBLIC_*` are baked in at build time**, so after setting them you must **redeploy** on
> Vercel (Deployments → ⋯ → Redeploy) for them to take effect. If `NEXT_PUBLIC_SITE_URL` is left
> unset it silently falls back to `localhost:3000`, which breaks your sitemap and share cards.

## 4. Update the Google OAuth client

In Google Cloud Console → Credentials → your OAuth client, add:
- **Authorized redirect URIs**: `https://ecoexpress-api.onrender.com/login/oauth2/code/google`
- **Authorized JavaScript origins**: `https://ecoexpress-nu.vercel.app` and `https://ecoexpress-api.onrender.com`

(Keep the localhost entries too, so local dev still works.)

## 5. Set up the Razorpay webhook — this makes payments settle

1. Razorpay Dashboard → Settings → **Webhooks** → **Add New Webhook**
2. URL: `https://ecoexpress-api.onrender.com/api/v1/payments/webhook`
3. Active events: `payment.captured`, `payment.failed`, `refund.processed`
4. Set a **secret** (any strong string). Copy it into `RAZORPAY_WEBHOOK_SECRET` on the API and
   save/restart.

Now a completed payment fires `payment.captured` → your API verifies the signature and amount →
the order flips `PENDING_PAYMENT → PAID`. (This is the piece that can't work on localhost without a
tunnel — on Render it just works.)

**How anyone knows a payment worked (test mode):** the checkout and order pages show a "Test
payment mode" note with the card to use. Enter card **`4111 1111 1111 1111`**, any future expiry,
any CVV, then choose **Success** on the simulated bank page (or pay by UPI `success@razorpay`). The
order then flips to **PAID** and the invoice becomes available — so a recruiter can see a full,
real-looking transaction without spending money. This note is controlled by
`NEXT_PUBLIC_PAYMENTS_TEST_MODE` and disappears when you go live.

## 6. Verify

- `https://ecoexpress-api.onrender.com/api/v1/ready` → `{"status":"ready",...}` (this is the
  endpoint the storefront's red→green "waking up" bar polls — expect a wait on the very first hit
  after the service has slept)
- `https://ecoexpress-api.onrender.com/actuator/health` → `{"status":"UP"}`
- Open the storefront, sign up → verification email arrives.
- Sign in with Google → lands back logged in.
- Add to cart → checkout → pay with test card `4111 1111 1111 1111` → order shows **PAID** shortly
  after (webhook), and the confirmation email arrives.
- Sign in as your bootstrap admin → the **Admin** link appears → dashboard loads.

## Notes & gotchas

- **Free-tier cold starts.** Free services sleep after ~15 min idle and take ~50s to wake. A webhook
  hitting a sleeping API is delayed, but Razorpay retries, so payments still settle. Use a paid
  instance for real traffic.
- **First deploy is slow.** The API image builds Maven from scratch (~3–5 min); later deploys are
  cached.
- **Caching.** Default is in-process (Caffeine) — fine for one instance. To use Upstash Redis, set
  `SPRING_PROFILES_ACTIVE=google-oauth,prod` and add `REDIS_HOST/REDIS_PORT/REDIS_PASSWORD`.
- **R2 must stay reachable.** Product images/certs are served from the R2 public URL; keep the
  bucket's Public Development URL (or a custom domain) enabled.
- **Test vs live payments.** Test keys work immediately. Real money needs Razorpay account
  activation (KYC) — the `onrender.com` URL satisfies their "website" requirement; no custom domain
  needed.
- **Own accounts before launch.** Google OAuth client, Resend domain, R2 bucket, and the Gemini
  key are currently borrowed/shared from the Candor app. Register dedicated ones before real
  customers. The Gemini free tier is **20 requests/day** — AI features (meal planner, fridge scan,
  recipe suggestion, product-content generation) will hit that quickly; the UI shows an "AI is over
  its limit" banner when they do. Use a billing-enabled Gemini key for a live demo.
- **Demo login.** The login page has one-click **"As Admin" / "As User"** buttons (controlled by
  `NEXT_PUBLIC_ENABLE_DEMO_LOGIN`). Great for recruiters, but "As Admin" grants full admin access and
  bakes the admin password into the public bundle — turn it off (or keep only "As User") for any
  deployment holding real data.
- **Structured logs.** Add `json` to `SPRING_PROFILES_ACTIVE` (e.g. `google-oauth,json`) for
  one-JSON-object-per-line logs to stdout (Render captures stdout; aggregators can parse it). Every
  log line carries an `X-Request-Id` correlation id. `prod` also enables JSON but additionally
  switches caching to Redis (needs `REDIS_*`).
